# ACGN 每日资讯站 · 建站方案书

本文回答三个问题：站点覆盖什么（范围与信息源）、怎么做出来（技术路线与流水线）、怎么长期活下去（合规、成本与演进）。文末对应本仓库已实现的原型。

## 一、覆盖范围设计

ACGN 四个字母天然给出了栏目框架，但每个领域内部还要区分"资讯类型"，否则内容会杂乱。建议用两个维度组织内容：领域（A/C/G/N + 综合）作为主导航，资讯类型作为条目内的隐性属性。

动画（A）方向的核心资讯类型包括：新作发表与制作决定、每日/每周放送表、声优与制作人员情报、剧场版与活动信息、季度新番热度榜。放送表是每日更新站的"刚需内容"——它天然每天都有新东西，能保证站点日日有料。

漫画（C）方向包括：新连载与完结/休载消息、单行本发售日、平台更新（JUMP+、MangaDex、国内平台）、漫画奖项与销量榜。

游戏（G）方向在 ACGN 语境下应偏向二次元游戏生态：主机/PC 新作与发售日、手游开服与版本更新、Galgame/视觉小说、独立游戏、销量榜（Fami通周榜）。综合游戏媒体内容量很大，需要靠关键词或来源筛选，避免游戏区淹没其他三区。

小说（N）方向是四者中信源最稀缺的：轻小说新刊速报、文库动态、Web 小说平台（成为小说家吧、Kakuyomu）榜单、轻改动画消息。正因稀缺，做好这一块反而是差异化竞争点。

综合区兜底：漫展与线下活动档期、周边与联名、业界人事与财报、跨领域企划（如"某 IP 动画化+游戏化同时公布"）。

另一个必须提前决定的范围问题是语言与地域：日文源最快最全、英文源（ANN 等）编辑质量高、中文源（机核、触乐、萌百）有本地视角但快讯少。建议三语并收、界面标注语种，后期再加机器翻译层。

## 二、信息源清单（截至 2026 年中可用性核查）

需要正视一个现实：中文 ACG 领域的 RSS 生态已严重萎缩，多数中文源要靠 RSSHub 这类"万物转 RSS"工具间接获取；而结构化 API 反而是最稳定的每日内容来源。因此推荐"API 打底 + RSS 补充"的组合。

| 来源 | 类型 | 领域 | 语言 | 说明 |
|---|---|---|---|---|
| AniList GraphQL | API（免密钥） | 动画 | 多语 | 单端点 graphql.anilist.co，可查放送表、作品、制作组，本站用它生成"今日放送" |
| Jikan (api.jikan.moe) | API（免密钥） | 动画/漫画 | 英 | MyAnimeList 非官方 REST 镜像，季度新番、热度榜、档期 |
| Bangumi API (api.bgm.tv) | API（免密钥） | 动画 | 中 | /calendar 返回每日放送且带中文条目名，中文站刚需 |
| MangaDex API | API（免密钥） | 漫画 | 多语 | 最新章节、按翻译语言过滤，漫画区的稳定日更源 |
| Anime News Network | RSS + 自有API | 动画 | 英 | 老牌动画新闻，RSS 稳定，另有 XML 报告 API |

## 三、技术路线对比与选型

做一个每日更新的资讯站，主流路线有四条，按运维复杂度从低到高排列。

路线一：静态站 + GitHub Actions 定时聚合（本原型采用）。定时任务跑 Python 脚本抓取数据写成 JSON，静态页面读 JSON 渲染，GitHub Pages 托管。优点是零服务器、零成本、数据即 Git 历史（天然归档与回滚）、故障面极小；缺点是更新频率受限（分钟级实时性做不到，但"每日"绰绰有余）、无法做用户系统。同类项目（如聚合 80+ 源的 AI 资讯聚合器、garss）已验证这条路线可长期稳定运行。

路线二：Serverless（Cloudflare Workers/Pages + KV/D1 + Cron Triggers）。适合想要更高更新频率（小时级）和边缘加速的场景，仍然接近零成本，但调试比路线一复杂。是路线一的自然升级目标。

路线三：动态站（VPS + FreshRSS/TTRSS + 自建前端，或 Node/Python 后端 + 数据库）。适合需要用户订阅、评论、个性化推荐时再上。月成本 5-10 美元起，需要自己管服务器与备份。

路线四：现成 CMS（WordPress + RSS 聚合插件）。上手最快但同质化严重、性能与安全维护成本高，不推荐作为长期方案。

选型结论：从路线一起步，信息源和内容形态验证成熟后，视需求迁移到路线二或三。前端刻意做成"读 JSON 的单文件页面"，就是为了迁移时后端随便换、前端不用动。

## 四、每日流水线设计（原型已实现部分 ✓）

整条流水线是：抓取 → 规范化 → 分类 → 去重 → 排序 → 输出 → 发布 → 归档。

抓取层（✓）：RSS 用 feedparser 统一解析；API 各写一个适配器函数（AniList 用 GraphQL 查今日放送、Jikan 查季度热番、Bangumi 查每日放送、MangaDex 查最新章节并带出作品名）。各源域名互不相同，因此用线程池并发抓取（并发度可配，默认 4），整体耗时从串行的分钟级降到几十秒；HTTP 层带 429/5xx 自动退避重试，并发阶段失败的源在冷却后还会串行补捞一轮，兜住瞬时故障。任何单源失败只记警告不中断整体，HTTP 错误页与 0 条返回都会显式告警而非静默吞掉，每源健康状况随 latest.json 的 sources 字段输出，供人工巡检。设置 UA 标识与超时，做有礼貌的抓取方。

规范化（✓）：所有条目压成统一结构 {id, title, url, source, category, lang, weight, published, summary}，摘要去 HTML 标签并截断 160 字——只留"引子"，全文点回原站，这既是产品选择也是版权红线。

分类（✓）：源级预设分类为主，综合源的条目再走关键词表兜底归类（"剧场版/声優"→动画，"Steam/发售"→游戏……）。关键词表放在配置文件里随时可调。升级方向是把兜底分类换成调用 LLM 做零样本分类，顺带生成一句话中文摘要。

去重（✓）：标题归一化（去空白标点、统一大小写）后取 MD5 指纹，跨源同题只保留权重最高的来源。升级方向是 SimHash/向量相似度，处理"同一新闻不同措辞"。

排序与输出（✓）：按源权重和发布时间排序，截断到每日上限，写出 latest.json、当日归档 archive/YYYY-MM-DD.json 与归档索引。

发布与调度（✓）：GitHub Actions cron 每天北京时间 6:00 与 18:00 各跑一次（注意 cron 写的是 UTC），数据提交回仓库后自动部署 Pages。保留 workflow_dispatch 手动触发入口便于调试。

归档（✓）：默认保留 90 天，前端提供"往期"下拉直接翻历史。因为归档就是 Git 提交，实际上全部历史永久可溯。

RSS 输出（✓）：聚合结果同时吐出根目录 feed.xml（RSS 2.0，取权重最高的 50 条，guid 用标题指纹保证订阅端去重），页面 head 带 alternate 声明、页脚带订阅入口，别人可以直接订阅本站。

尚未实现、按优先级排列的增强项：英日文标题机翻/LLM 翻译成双语显示；每日头条摘要（LLM 从当日条目中挑 5 条写导语）；站内订阅推送（Telegram/邮件）；全文搜索（当归档量大后用 pagefind 做纯静态搜索）。

## 五、前端设计说明

定位是"30 秒扫完今日四象限"的日报，而非阅读器。因此采用日刊放送表式的信息密度：报头 + 分类统计条 + 单列条目流，每条只有色块徽章（動/漫/遊/小/綜）、标题、来源、时间、语种五个元素。四个领域各绑定一个频道色（朱/蓝/绿/紫）贯穿徽章与导航。签名元素是左侧竖排书脊式分类导航（移动端自动横排）。整站无框架无构建，单 HTML 文件，示例数据内置兜底，保证"clone 下来双击就能看"。

## 六、合规、成本与风险

版权层面守住三条线：只收标题+链接+短摘要并标注出处（聚合行为的通行安全区）；不转载全文、不热链图片；对方若声明禁止聚合或来函要求下架，立即移除该源。抓取层面遵守 robots.txt、控制频率、带可识别 UA。若未来引入用户投稿或评论，需另行处理内容审核。

成本：路线一为零现金成本（公开仓库的 Actions 与 Pages 免费额度远超本项目用量，每日两次运行约消耗 4-6 分钟 Actions 时长）。若加自定义域名约 60 元/年；自部署 RSSHub 可用免费档 Serverless 平台承载。

主要风险与对策：信息源失效（RSS 停更、API 改版）——脚本已做单源隔离，另建议每月人工巡检一次抓取日志；RSSHub 公共实例限流——中期自部署；GitHub 在部分网络环境访问不稳——可把 Pages 换成 Cloudflare Pages 同构部署；内容同质化——把稀缺的 N（轻小说）板块和"每日放送表"做深，是与通用动漫站拉开差距的抓手。

## 七、三个月演进路线建议

第一个月跑通基本盘：部署原型、巡检各源实际抓取质量、按口味增删 feeds.yml、微调关键词分类表。第二个月做内容增强：接入 LLM 做双语标题与每日五条头条导语、输出自己的 RSS。第三个月做分发与留存：Telegram 频道自动推送、归档搜索、可选的自部署 RSSHub 扩充中文源。到那时如果日活验证了需求，再考虑迁移到带用户系统的动态架构。
